# EV充電器 営業商談用クロージング支援ツール「EV-Insight」要件定義書 v2.0

## 1. プロジェクト概要

EV充電器の導入を検討している企業・商業施設に対し、営業が商談の場で「競合状況」「補助金」「投資回収（コストカバー）」の3点から論理的に導入を正当化し、稟議の壁を突破するためのWebダッシュボード。

※不確かな推計データ（人流など）や、過度な売上アップ予測は排除し、堅実で反論不可能なファクトベースのツールとする。

---

## 2. 解決する「本当の」課題

### 課題①「近隣と競合しないか？本当に需要はあるのか？」
- **解決**: 地図上で「普通充電（滞在型）」と「急速充電（通過型）」を明確に色分けし、用途の棲み分けを視覚的に証明する。

### 課題②「設備投資とランニングコスト（基本料金等）で赤字にならないか？」
- **解決**: 補助金を適用した場合の初期費用と、月額コストを最低限回収するための「超・保守的な損益分岐点（最低必要稼働数）」をシミュレーションする。

### 課題③「商談中のアプリ操作でもたつく・エラーになる」
- **解決**: 商談前に営業がシミュレーション条件をセットし、その状態のURLを発行して商談に臨める「事前セットアップ機能」を実装する。

---

## 3. コア機能要件

### ① 競合・共存マップビュー（Google Maps API 等）
**機能:**
- 対象施設を中心にピンを立てる。
- 周辺の充電器をAPI等で取得・表示する際、「急速充電（赤ピン）」と「普通充電（青ピン）」を**絶対に色分けして表示**する。
- 周辺の大型施設（競合になり得るホテルや商業施設）のピンを立て、充電器の有無を可視化。

**目的:** 「近くのコンビニに急速があるから不要」という反論に対し、「貴社のような滞在型施設には普通充電の青ピンが必要です。このエリアにはまだ青ピンが足りていません」と説明するため。

### ② 防御型・損益分岐シミュレーター
**機能:** 売上アップの夢を見せるのではなく、「赤字にならないライン」を証明する電卓UI。

**入力項目:**
- 導入予定の充電器の種類と基数
- 補助金の有無（トグルスイッチ）
- 予想される月額の電気代基本料金アップ分・保守費用
- 1回あたりの充電料金設定 / 施設利用の客単価

**出力項目（自動計算）:**
- 「月額コストを相殺するために必要な、1日あたりの最低充電回数（または最低集客数）」

**目的:** 「1日にたった〇台充電されれば（または〇人が施設を利用すれば）、コストは完全にペイします」という極限までハードルを下げた数字を提示し、決裁者の恐怖を払拭する。

### ③ ファクトデータ・ダッシュボード（マクロ情報）
**機能:**
- 国や自治体の「CEV補助金」の予算消化率や締め切り関連の最新ニュース（NewsAPIや手動更新データ）。
- 自販連等が公表している、確実な「都道府県別」のEV・PHEV販売台数推移（モックCSVで実装）。

**目的:** 曖昧な推計データではなく、誰も反論できない「公的なファクト」と「補助金終了のタイムリミット」だけを提示し、「今すぐやる理由」を作る。

## 4. デザイン要件

商談の場で決裁者に見せても「安っぽくない」「説得力のある」見栄えにする。Claude Design の思想（暖かみのあるニュートラル、上質なタイポグラフィ、過度な装飾を排した余白）を踏襲する。

### 4.1 デザイン原則
- **Premium / Trustworthy**: 数字を主役に。決裁者に出して恥ずかしくない品位。
- **Calm UI**: 派手なグラデやネオン色は禁止。情報の階層を余白とフォントウェイトで作る。
- **One screen, one message**: 各画面は商談での1メッセージ（例:「赤字にならない最低ライン」）に集中させる。
- **Numbers first**: KPI は tabular-nums で大きく、ラベルは控えめに。
- **Quiet motion**: トランジションは 150〜250ms、ease-out。バウンスや弾みは入れない。

### 4.2 ビジュアル仕様
- **配色**:
  - ベース: 暖色系オフホワイト（`#FAF9F5` 系）/ ダークは `#1F1E1B` 系
  - アクセント: Anthropic 系コーラル/オレンジ（`#D97757` 付近）を「行動・確信」に限定使用
  - ステート: 損益分岐 OK = エメラルド系 / 警告 = アンバー / 危険 = ローズ
- **タイポグラフィ**:
  - 見出し: Geist / Inter のディスプレイ寄り、トラッキング tight
  - 本文: Geist Sans, 14–15px
  - 数値: tabular-nums 必須、見出しサイズで使う
- **レイアウト**:
  - 最大幅 1200px、左右 24–32px パディング
  - カード間ギャップ 24px、セクション間 48–64px
  - 角丸 `rounded-xl` 統一、ボーダーは薄色1pxのみ。重いシャドウは禁止。
- **コンポーネント方針**: shadcn/ui を素のまま使わず、トーン（border、shadow、padding）を統一する。

### 4.3 必須UI要素
- グローバルヘッダー（ブランド + ナビ + 「商談用URLをコピー」CTA）
- 各画面のヒーロー（数字 1個 + 1行の説明）
- KPI カード（数値・単位・補足テキストの3層）
- フッター（出典・最終更新日）

---

## 5. 推奨技術スタック

- **フロントエンド**: Next.js (App Router), React, Tailwind CSS, shadcn/ui
- **マップ描画**: Google Maps API（マーカーのカスタムアイコンによる色分けが必須）
- **状態管理/共有**: URLクエリパラメータ（`?lat=xxx&lng=yyy&cost=zzz` のようにURLに状態を持たせ、営業が商談前にURLを生成してブックマークできるようにする）
- **グラフ・UI**: Recharts, Radix UI

---

## 6. 開発フェーズ

### Phase 1: 防御型・損益分岐シミュレーターの構築
マップより先に、入力値に対して「1日あたりの最低必要充電回数」を算出する計算ロジックとUI（shadcn/ui）を作成する。

### Phase 2: 状態のURLパラメータ化
Phase 1で作ったシミュレーターの数値を、URLパラメータに同期させる仕組みを作る（リロードしても数値が保持されること）。

### Phase 3: 競合・共存マップの統合
Google Maps等を組み込み、ダミーデータで構わないので「急速充電（赤）」と「普通充電（青）」のピンを描き分ける。
